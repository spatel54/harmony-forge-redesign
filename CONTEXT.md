## Ontology

| Term | Definition |
|---|---|
| **HarmonyForge** | An ante-hoc "Glass Box" co-creative system for SATB music arrangement |
| **Tactile Sandbox** | The primary editing screen — symbolic score rendering (VexFlow) + playback (Tone.js) |
| **Theory Inspector** | The sidebar UI — LLM-driven Glass Box reasoning and Red Line violation flags |
| **Glass Box** | The design principle: all reasoning is visible and anchored in academic sources, never opaque |
| **Intrinsic Determinism** | Every claim is anchored to an academic definition, not probabilistic folk knowledge |
| **Red Line** | Inline violation annotations on the VexFlow canvas (parallel fifths, voice crossings, etc.) |
| **SATB** | Soprano, Alto, Tenor, Bass — the four-voice arrangement format this system specializes in |
| **Symbolic** | Music represented as pitch/duration/voice data (MusicXML/JSON) — never raw audio or waveforms |
| **Nocturne** | The dark color palette / theme for HarmonyForge |
| **Sonata** | The light color palette / theme for HarmonyForge |
| **Superman** | The human owner of this project; the final authority on all decisions |
| **Ralph Wiggum Loop** | The autonomous execution pattern: loop.sh → PROMPT_build.md → one task → backpressure → commit |
| **Backpressure** | `npx tsc --noEmit` + `npm run build` must pass before every commit; failure is a hard stop |
| **FIC** | Frequent Intentional Compaction — the upstream planning workflow (research → plan → implement) |
| **ADR** | Architecture Decision Record — permanent, immutable log of why a technical decision was made |
| **Spec** | JTBD specification document — observable user outcomes with testable acceptance criteria |
| **JTBD** | Job to Be Done — the user outcome framing used inside every spec |

## Patterns

- **Three Gears**: Specification (Gear 1) → Planning with planner (Gear 2) → Execution via loop (Gear 3)
- **HCI Protocol**: Every interactive session follows preview → gate → execute (`.claude/rules/hci-protocol.md`)
- **Theory Named Strategy**: Every music theory claim is anchored — "X is defined as [definition], per [source]"
- **Architecture Boundary**: Frontend renders/schedules/surfaces; constraint logic lives in the backend only
- **Atomic Design**: UI components are organized as atoms → molecules → organisms

---

## Policies

- **Never run `loop.sh` on `main` or `master`** — feature branches only
- **Never introduce ad-hoc hex color values** — Nocturne/Sonata tokens only
- **Never implement constraint-satisfaction or violation-detection logic in the frontend** — backend only
- **Never edit the rationale, options, or consequences of an Accepted ADR** — supersede instead
- **Always run `npx tsc --noEmit` before committing**
- **Known pre-existing issue:** `npm run lint` fails with `zod/v4/core` module error (Node 25 / eslint-config-next 16.1.6 incompatibility) — use `npx tsc --noEmit` + `npm run build` as authoritative backpressure until resolved

## CONTEXT.md Maintenance

Every directory in this repo contains a `CONTEXT.md` that records its purpose, key files, patterns, and decisions. These files are the durable context that survives every context-window reset — stale entries feed false premises to the loop.

**Enforcement layers:**

1. **AGENTS.md rule** — after every task, the loop must update the CONTEXT.md in every directory it touched. This is treated as a backpressure step.
1. **SpecStory distillation script** — run at the start of any new session to pull last session's decisions into the relevant CONTEXT.md files automatically:

```bash
bash scripts/specstory-to-context.sh
```

Flags: `--dry-run` (preview without writing) · `--since YYYY-MM-DD` (limit to sessions after a date).

1. **PostToolUse hook** (`.claude/settings.json`) — surfaces a reminder after file edits in interactive sessions.

**Script internals:** reads `.specstory/history/*.md`, calls `claude -p` to extract a structured JSON summary (slug · directories touched · insights), then appends a row to the `## Session Insights` table at the bottom of each relevant CONTEXT.md. Processed session IDs are tracked in `.specstory/.processed` for idempotency.
